05 — How It Works

Anatomy of an MCP App

01
Create a normal MCP server with two key capabilities: tools and resources
02
A tool (e.g. show_demo_app) responds with _meta.ui.resourceUri pointing to a ui:// URI
03
A resource at that URI returns HTML with MIME type text/html;profile=mcp-app
04
The HTML is a regular web page — vanilla HTML/CSS/JS, no special framework needed
05 — How It Works

The Flow

1
User says something like "show me the demo app"
2
Goose calls the MCP tool → server responds with a ui:// resource URI
3
Goose fetches the HTML from that resource and renders it in a sandboxed iframe
4
The app communicates back via window.postMessage using JSON-RPC
05 — How It Works

How MCP Apps Talk to Goose

ui/initialize
App → Host
App calls on load to get host context (theme, window size)
ui/message
App → Host
App sends a message back into the chat (user intent → agent)
ui/notifications/host-context-changed
Host → App
Goose pushes theme changes (dark mode, light mode) to the app
ui/notifications/size-changed
App → Host
App tells Goose its height so the iframe resizes dynamically
06 — Security

The Elephant in the Room

Running someone else's HTML inside your desktop app is basically saying "here, execute this code on my machine."

The original experiment was called MCP-UI. Everyone raised concerns: malicious scripts, data exfiltration, all from inside your chat window.

06 — Security

Security Constraints

✓ Apps run in a sandboxed iframe with a strict Content Security Policy
✓ By default: same-origin only, no external network, no nested iframes
✓ Explicitly allowlist domains for fetch, CDN resources, or embedded iframes via a csp config
✓ Only add domains you trust — keep the list minimal
Principle: deny by default, allowlist explicitly.
